# /phase-verify — 세션 문서 그대로 실행·재현 (참가자 시뮬레이션)

처음 보는 참가자가 `docs/sessions/0N-*.md` **문서 하나만** 보고 따라 했을 때, 문서가 보여주는 코드를 순서대로 만들고 문서에 적힌 명령을 그대로 실행하면 끝까지 동작하는지를 **Claude 가 실제로 그 참가자가 되어 수행**해 확인한다. CLAUDE.md §11(챌린지 참가자 관점 우선)의 자동 강제 도구.

> 인자 `N` 은 **세션 번호**다 (`/phase-verify 4` → `docs/sessions/04-async-ingestion.md`). 명령 이름은 `phase-verify` 지만 피연산자는 세션 문서임에 유의.

## 핵심 원칙 (이 순서가 곧 설계)

1. **문서가 유일한 소스.** 코드는 문서가 보여주는 내용을 **그대로** 작성하고, 명령은 문서에 적힌 블록을 **그대로** 실행한다. 변수·플래그·순서를 임의로 바꾸지 않는다.
2. **`save-points/session-NN/complete/` 는 정답지(oracle)이지 소스가 아니다.** 거기서 코드를 베껴오지 않는다. 각 단계가 끝난 뒤 *대조용*으로만 diff 한다.
3. **문서대로 했는데 정답지와 어긋나거나, 명령이 막히면 = 문서 결함.** 모든 어긋남·막힘은 §11 결함으로 기록한다(어느 단계, 어떤 OS 변형, 무엇이 빠졌는지, 문서 줄/섹션).
4. **크로스 환경.** 챌린지 참가자는 macOS · Linux · Windows(PowerShell/WSL) 모두를 포함한다. 문서의 모든 명령 블록에 대해 **OS 커버리지 검사**를 수행한다.
   - **듀얼 커버가 필요한 블록**: 셸 문법·환경에 의존하는 블록은 반드시 bash 변형 + PowerShell 변형을 모두 제공해야 한다. 필요 여부 판단 기준:
     | 패턴 | bash | PowerShell | 주의 |
     |---|---|---|---|
     | 환경변수 설정 | `export VAR=val` | `$env:VAR = "val"` | 반드시 듀얼 |
     | 환경변수 참조 | `$VAR` | `$env:VAR` | az CLI 인자 안에서도 다름 |
     | 줄 연속 | `\` | `` ` `` (백틱) | 혼용 시 상대 OS 에서 파싱 오류 |
     | 파일 경로 | `/` | `\` 또는 `/` | az CLI·Git은 `/` 허용, 일부 도구는 `\` 필요 |
     | 디렉터리 이동 | `cd path` | `Set-Location path` | 동일 결과이나 형식 다름 |
     | 파일 복사 | `cp -a src/. dst/` | `Copy-Item -Path src/* -Destination dst -Recurse -Force` | 반드시 듀얼 |
     | 변수 치환(인라인) | `$(cmd)` | `$(cmd)` | PowerShell 도 동일 — 듀얼 불필요 |
     | az CLI / docker / git | 동일 | 동일 | 듀얼 불필요 (환경변수 참조 부분만 확인) |
   - **듀얼이 없는 쪽 = OS 한쪽만 결함.** bash 블록만 있고 PowerShell 변형이 없거나, 반대도 마찬가지면 `OS 한쪽만` 결함으로 기록.
   - **실행**: Claude 는 현재 환경(Linux/macOS)의 bash 변형을 실행한다. PowerShell 변형이 문서에 있으면 구문 정확성을 육안 검토해 결함 여부를 판정(실제 실행 불필요).
   - **az CLI 블록에 섞인 환경변수 참조**: 예) `--parameters userObjectId=$OID` 는 bash에서만 동작한다. PowerShell 에서는 `$env:OID` 또는 변수 할당 후 참조가 필요하므로, az 블록 안에 bash 전용 `$VAR` 참조가 포함된 경우도 결함으로 잡는다.
5. **배치(placement) 명확성 — 어느 코드를·어느 파일/터미널에·어디에.** 문서가 코드·명령을 보여주는 것만으로는 부족하다. 처음 보는 참가자가 헷갈리지 않으려면 문서가 다음을 **명시적으로** 안내해야 한다. 안내가 없거나 모호하면(추론·관행에 의존해야 하면) 그 자체가 §11 결함이다:
   - **코드 블록 → 어느 파일**: "이 내용을 `<정확한 상대경로 파일>` 에 작성/추가하라"가 있는가. 파일명·경로가 빠졌거나, 새 파일인지 기존 파일 수정인지 불명확한가.
   - **파일 내 위치**: 기존 파일을 수정한다면 "어디에"(맨 끝 추가 / 특정 함수·블록 교체 / import 구문 추가 등)가 드러나는가. 전체 파일을 다시 보여주는지, 일부만 보여주면서 삽입 위치를 지시하는지.
   - **명령 블록 → 어느 터미널·어느 작업 디렉터리**: 그 명령을 **어디서**(repo 루트 / `workshop/` / `apps/api/` 등) 실행하는지, 어떤 셸·도구 컨텍스트(로컬 셸 / `az` 로그인된 셸 / `psql` 프롬프트 안 / `uv run` 등)인지 드러나는가. 여러 터미널·컨텍스트를 오가는 세션이면 "지금 어느 터미널인지" 전환이 추적 가능한가.
   - 판정 기준: **참가자가 "이걸 어디다 넣지?/어디서 치지?" 를 스스로 추측해야 하면 결함.** Claude 본인이 정답지·repo 구조를 알기에 자연히 맞춘 경우라도, 문서 텍스트만으로는 모호했다면 결함으로 잡는다.

## 사전 조건

- **세션 번호 N** 확인(인자 또는 본문). 누락 시 짧게 묻고 중단.
- `docs/sessions/0N-*.md` 존재 확인. (구 `docs/learning-paths/` 경로는 폐기됨 — 실제 경로는 `docs/sessions/`.)
- **실제 Azure 배포가 일어난다.** `az account show` 로 로그인·구독 컨텍스트 확인. CLAUDE.md §7 대로 **유료 자원이 뜨므로**, 시작 전 이 세션이 띄우는 주요 유료 자원을 한 줄로 고지한다.
- **작업 폴더 준비.** 문서의 `## 시작본 코드 받기` 절을 그대로 실행해 `workshop/` 을 만든다 (`cp -a save-points/session-NN/start/. workshop/`). 깨끗한 재현을 위해 기존 `workshop/` 이 있으면 덮어쓸지 사용자에게 먼저 confirm.

## 실행 범위 — phase-verify 는 문서의 명령을 직접 실행한다

이 Skill 은 **참가자 시뮬레이션**이다. 참가자는 문서의 명령을 자기 손으로 실행하므로, Claude 도 이 Skill 안에서는 **문서(`docs/sessions/0N-*.md`)에 적힌 그대로의 명령을 직접 실행**한다 — what-if·배포·빌드/푸시·호출 검증·배포 후 확인 조회까지. 사용자가 `/phase-verify` 를 명시 호출한 것이 그 실행 맥락이다.

- **문서에 적힌 명령에 한정된다.** 권한은 "문서가 보여준 명령"에만 적용된다. 문서에 없는 명령을 임의로 추가·대체하지 않는다(가드 참고). 부득이한 보정은 "문서 외 보정"으로 표시한다.
- **삭제는 실행하지 않는다.** `az ... delete` 류는 `block-az-delete.sh` hook 이 차단한다(정상). 이 Skill 은 생성·배포·확인만 한다. 정리는 `/phase-cleanup` 의 몫.
- **권한 차단 시.** 환경 권한 설정으로 특정 명령이 막히면, 그 명령을 사용자에게 그대로(문서의 블록 그대로) 전달해 직접 실행하도록 안내하고, 결과를 받아 검증을 이어간다. 권한 설정 변경은 사용자가 직접 한다.
- **셸 변수 인라인.** 문서가 `OID=$(...)` 처럼 한 블록에서 변수를 만들고 다음 블록에서 `$OID` 를 쓰면, Bash 호출 간 셸 상태가 유지되지 않으므로 **그 변수를 쓰는 명령과 같은 한 번의 Bash 호출에 합쳐** 실행한다(예: `--parameters userObjectId=$(az ad signed-in-user show --query id -o tsv)`). 실행 단위만 합치는 것이며 플래그·값은 바꾸지 않는다 — "문서 외 보정"이 아니다.

## 실행 절차 — 문서 단계를 위에서 아래로

세션 문서의 `## N 단계 : ...` 헤딩을 **순서대로** 진행한다(세션마다 단계 구성은 다름 — 문서에 있는 단계를 그대로 따른다). 각 단계마다 A→D:

### A. 단계 읽기
해당 `## N 단계` 블록 전체(하위 `###` 포함)를 Read. 그 안에서 (a) **생성/수정할 파일과 그 내용**, (b) **실행할 명령 블록**, (c) **문서가 명시한 "기대 결과"**, (d) **배치 안내**(각 코드 블록이 어느 파일·파일 내 어디에 들어가는지, 각 명령 블록을 어느 터미널·작업 디렉터리·컨텍스트에서 실행하는지)를 식별한다. 코드/명령 블록마다 (d)가 **문서 텍스트에 명시돼 있는지** 체크 — 본인이 repo 구조를 알아서 채운 부분은 "문서엔 없었다"로 표시(핵심 원칙 5).

### B. 코드 생성 단계 (예: "Bicep 모듈 조립", "앱 코드 채우기")
- **배치 검증 먼저.** 코드 블록을 쓰기 전에, 문서가 (1) **어느 파일**에 넣으라고 했는지, (2) 기존 파일이면 **파일 내 어디에**(전체 교체 / 끝에 추가 / 특정 블록 교체 / import 추가) 넣으라고 했는지 명시했는지 확인. 문서에 파일명·경로가 없거나, 새 파일/기존 수정 구분이 모호하거나, 일부 코드만 보여주면서 삽입 위치를 안 짚으면 → **배치 결함**으로 기록(어느 블록, 무엇이 모호한지, 줄/섹션).
- 문서가 "`<파일>` 에 이 블록을 채우라" 고 하면, Write/Edit 로 그 파일에 문서가 보여준 내용을 **그대로** 작성한다. 정답지에서 가져오지 않는다.
- 저장 시 PostToolUse lint hook(`bicep-build.sh`, `python-lint.sh`)이 자동 실행된다. **lint 실패 = 문서가 보여준 코드가 통과 못 한다는 결함** → 기록.
- 단계 종료 후 `save-points/session-NN/complete/` 의 대응 파일과 `diff`. **의미 있는 차이 = 문서가 그 코드 일부를 안 보여줬다는 신호** → 결함으로 기록하고, 재현을 이어가기 위해 정답지 기준으로 보정한 뒤 **"문서 외 보정"임을 명시**(그래야 결함이 가려지지 않음).

### C. 명령 실행 단계 (예: "Bicep 배포", "시드", "빌드·배포·호출")
- **배치 검증 먼저.** 명령 블록마다 문서가 (1) **어느 작업 디렉터리**(repo 루트 / `workshop/` / `apps/api/` 등)에서, (2) **어느 터미널·컨텍스트**(로컬 셸 / `az` 로그인 셸 / `psql`·`redis-cli` 프롬프트 안 / `uv run --project ...` 등)에서 치라고 했는지 명시했는지 확인. 한 단계가 여러 터미널·컨텍스트를 오가면 문서가 그 **전환을 명시**(예: "새 터미널에서", "psql 프롬프트에서 빠져나와")하는지 본다. 빠졌거나 모호하면 → **배치 결함**으로 기록.
- **크로스 플랫폼 스캔.** 각 명령 블록에 대해 핵심 원칙 4 의 표를 적용해 OS 커버리지를 점검한다.
  1. 블록 언어 태그 확인 — ` ```bash ` / ` ```powershell ` / ` ```text ` 등.
  2. bash 블록 안에 `export`, `$VAR`(bash 변수), `\` 줄 연속, `cp -a` 등 **bash 전용 패턴**이 있는지 확인. 있으면 대응 PowerShell 블록이 함께 있는지 확인.
  3. PowerShell 블록 안에 백틱 줄 연속(`` ` ``), `$env:VAR`, `Set-Location`, `Copy-Item` 등 **PowerShell 전용 패턴**이 있는지 확인. 있으면 대응 bash 블록이 있는지 확인.
  4. az CLI 블록에 `$VAR`(bash 변수) 또는 `$env:VAR`(PowerShell 변수)이 포함돼 있으면 단일 블록으로 한쪽 OS 에서만 동작하는 것이므로 → **OS 한쪽만** 결함.
  5. 결함이 있으면 기록 후 현재 환경(bash) 기준 보정 명령으로 계속 진행하되 "문서 외 보정" 표시.
- 문서의 bash 블록을 **그대로** 실행한다.
- **배포**: 문서가 what-if(예: `### 3.1 변경사항 미리보기`)를 먼저 시키면 그대로 what-if → 그다음 `az deployment ... create`. **문서 순서를 따르는 것 자체가 CLAUDE.md §6 의 "검토 후 배포"를 충족**한다. what-if 가 **예상 밖 삭제/대량 변경**을 보이면 즉시 멈추고 사용자에게 보고(문서대로인데 위험 신호 = 결함).
- **이미지 빌드·푸시**(§5 IaC 예외): `az acr login` + `docker build --platform linux/amd64` + `docker push` 를 문서대로. (ARM Mac 함정 회피용 `--platform` 이 문서에 없으면 결함.)
- **호출·검증 커맨드**(curl / psql / KQL 등): 문서가 보여준 **"기대 결과"와 실제 출력을 대조**. 불일치는 기록.
- **명령이 실패하면**: 문서만으로 막힌 지점 = **1급 결함**. 에러와 재현 명령을 인용해 기록. 진행을 위해 최소 보정이 필요하면 **무엇을 왜 했는지 명시**하고 잇거나, 사용자 판단이 필요하면 멈춘다.

### D. GUI 확인 단계 (예: "Azure Portal UI 에서 확인")
문서의 GUI 확인 단계는 **Portal·Foundry·브라우저에서 눈으로 보라**고 하지만, Claude 는 그 **확인 항목 하나하나를 동등한 CLI 명령으로 변환해 직접 검증**한다(§11 — 모든 참가자가 재현 가능한 방식 우선, scale-to-zero 등으로 GUI 가 비실용적인 경우 포함).

- **GUI 항목 → CLI 매핑이 기본이고 필수다.** "넘어가고 사용자에게 맡긴다"가 아니라, 문서가 GUI 로 확인하라는 **각 항목의 의미(무엇이 보여야 하는가)** 를 읽고, 그것을 검증하는 `az ... list`/`show`·KQL·`curl` 등을 실행해 **기대 상태와 실제 출력을 대조**한다. 대표 매핑:
  - "Portal 의 RG Overview 에 자원 X·Y·Z 가 보인다" → `az resource list -g <rg> -o table` 로 해당 자원 존재 확인
  - "Foundry/Cognitive 에 deployment 2개가 보인다" → `az cognitiveservices account deployment list -n <acct> -g <rg> -o table`
  - "IAM 의 Role assignments 에 역할 R 이 (없)있다" → `az role assignment list --scope <resourceId> --query "[].roleDefinitionName" -o tsv` 로 부여/미부여 확인
  - "App Insights/Logs 에 데이터가 들어온다" → KQL(`az monitor log-analytics query` 등) 로 행 수·필드 확인
- **CLI 로 정말 확인 불가능한 항목만** 체크리스트로 사용자에게 넘긴다(예: 순수 시각적 렌더링, 스크린샷 캡쳐 그 자체). 이때도 "왜 CLI 로 안 되는지" 한 줄을 단다.
- 각 GUI 항목에 대해 **CLI 검증함 / 사용자몫(이유)** 중 무엇이었는지 충실도 리포트에 남긴다.

## 진행 중 상태 보고
각 단계 끝에 한 줄 요약: `[단계 N] 코드 X파일 생성 · 명령 Y개 실행 — 통과 / 결함 Z건`. 누적 결함 목록을 계속 유지한다.

## 종료 — 충실도 리포트 (최종 산출물)

모든 단계를 마치면:

1. **단계별 결과 표**

   ```
   | 단계 | 한 일 | complete 일치 | 명령 통과 | 배치 명확 | 결함 |
   |---|---|---|---|---|---|
   | 1 Bicep 조립 | main.bicep 8블록 작성 | ✅ | — | ✅ | 0 |
   | 3 배포 | what-if→create | — | ✅ | ⚠ 디렉터리 미표기 | 1 |
   | 5 빌드·호출 | 이미지2 + curl | — | ❌ /chat 500 | ✅ | 1 |
   ```

   `배치 명확` 열은 핵심 원칙 5 의 결과 — 코드 블록의 파일·위치, 명령의 터미널·디렉터리 안내가 문서에 명시됐으면 ✅, 모호하면 ⚠ 와 무엇이 빠졌는지.

2. **결함 목록** — §11 분류로:
   - `빠진 코드` — 문서에 없어서 정답지 보정이 필요했던 부분
   - `깨진 명령` — 문서대로 실행했을 때 실패하는 명령
   - `OS 한쪽만` — bash/PowerShell 중 한쪽 변형만 있는 블록. 예시:
     - `export VAR=value` bash 블록만 있고 `$env:VAR = "value"` PowerShell 블록 없음
     - `cp -a src/. dst/` bash만 있고 `Copy-Item` PowerShell 없음
     - `az ... --parameters key=$VAR` 처럼 az CLI 블록 안에 bash 변수 참조만 있고 PowerShell 변수 참조 없음
     - `\` 줄 연속 bash 블록만 있고 PowerShell 백틱 변형 없음 (또는 반대)
   - `환경 의존(절대경로·기존자원·수동권한 전제)` — 특정 사용자 환경을 전제하는 부분
   - `기대결과 불일치` — 문서의 기대 결과와 실제 출력이 다른 경우
   - `배치 모호(어느 파일/파일내 위치/어느 터미널·디렉터리 불명확)` — 참가자가 추측해야 했던 배치

   각 결함에 **`docs/sessions/0N-*.md` 의 줄·섹션 + 제안 수정**을 단다. 배치 결함은 "참가자가 추측해야 했던 지점"을 구체적으로(예: "3.2 의 `app.py` 블록 — 새 파일인지 기존 수정인지 없음", "4.1 의 `psql` 명령 — 어느 터미널인지 미표기") 적는다.
3. **배포 상태 + GUI 항목 CLI 검증 결과**: 실제로 뜬 자원(`az resource list -g rg-ai200ws-dev`), 헬스(문서 범위 내 `curl /healthz` 등). 문서의 GUI 확인 단계 각 항목을 어떤 CLI 명령으로 검증했고 기대 상태와 일치했는지 표로 남긴다(§D). CLI 로 확인 불가해 사용자에게 넘긴 항목은 이유와 함께 명시.
4. **자원 정리는 이 Skill 의 책임이 아니다.** 검증 끝나면 사용자가 GUI 확인 후 "정리해" → `/phase-cleanup`. 검증 흐름에서 잠시 부여한 임시 권한·ingress external 토글 등은 같은 흐름에서 회수(§7 예외).

## 가드

- **문서에 없는 명령을 임의로 추가·대체하지 않는다.** 부득이한 보정은 전부 "문서 외 보정"으로 표시해야 결함이 은폐되지 않는다.
- 이 Skill 은 **생성·배포 전용** — 삭제 명령이 없다. `az ... delete` 는 `block-az-delete.sh` hook 이 차단(정상 동작).
- **실제 비용 발생**(§7). 시작 전 유료 자원 고지, 종료 시 사용자에게 정리 경로(`/phase-cleanup`) 안내.
- 사용자 식별 정보(IP·objectId)는 §8 대로 배포 시점 override — 문서가 파라미터 파일에 박아두라고 하면 그 자체가 결함.

## 출력 톤
한국어(§4). 단계 헤더로 진행을 명확히. 결함은 빨간 깃발로(이모지 X). **충실도 리포트가 이 Skill 의 최종 산출물**이며, 그 결함들이 곧 §11 문서 개선 작업의 입력이 된다.

- **종료 시 "확인할 지점" 정리는 필수.** 리포트 마지막에 사용자가 바로 검토할 수 있게 **문제 있어 보이는 지점**(깨진 명령·배치 모호·기대결과 불일치 등)을 한데 모아 우선순위순으로 제시한다. 각 항목은 `docs/sessions/0N-*.md` 의 줄·섹션 + 무엇이 문제인지 + 제안 수정 형식.
- **문제가 없으면 없다고 분명히 말한다.** 억지로 결함을 만들지 않는다 — 모든 단계가 문서대로 재현되고 배치도 명확했으면 "확인이 필요한 문제 없음 — 문서만으로 끝까지 재현됨" 으로 명시 종료.
